## Notes

### Vision et positionnement
Folia est un réseau social dédié aux plantes d'intérieur et au jardinage. L'objectif est de créer une communauté engagée autour du soin des plantes, avec des fonctionnalités d'identification, de partage et de gamification.

Différenciateurs clés identifiés :
- Identification par photo (IA) intégrée nativement
- Gamification poussée (badges, défis, streaks d'arrosage)
- Données botaniques riches issues d'une base catalogue vérifiée
- Communauté et partage de boutures / échanges entre membres
- Mode hors-ligne pour les fiches plantes

### Architecture technique
Choix fondamentaux arrêtés :
- **Monorepo** : Turborepo + pnpm workspaces
- **Langage** : TypeScript strict (`strict: true` + `noUncheckedIndexedAccess`)
- **Architecture** : DDD + Vertical Slice par feature, architecture hexagonale
- **Base de données** : PostgreSQL 16 avec Drizzle ORM
- **Framework HTTP** : Fastify avec validation Zod
- **Tests** : Vitest (unitaires) + Testcontainers (intégration)
- **Erreurs** : Package partagé `@folia/errors`

### Organisation des services (Bounded Contexts)
12 microservices identifiés, déployés progressivement :
- **V1** : catalogue, identity, collection, social, geo
- **V2** : gamification, identification, notification, search
- **V3** : marketplace, content, media
- **V4** : analytics, i18n, audit

Premier service implémenté : **catalogue** (plantes, espèces, taxonomie).

### Interface layer du service catalogue
Implémentée lors de cette session :
- Schémas Zod (`plant.schema.ts`)
- `PlantController` avec gestion d'erreurs centralisée
- Routes Fastify complètes (CRUD + publish/unpublish)
- Use cases `PublishPlantHandler` et `UnpublishPlantHandler`
- `PostgresSpeciesRepository`

### Base de connaissance Obsidian
Vault créé à `/opt/projects/folia-vault` avec la structure :
- `01-business/` — vision, modèle économique, concurrence
- `02-product/` — roadmap, features, user stories
- `03-domain/` — glossaire, DDD, bounded contexts, events
- `04-technical/` — stack complète, services, patterns
- `05-decisions/` — ADRs globaux et par service
- `06-meetings/` — comptes rendus (ce dossier)

## Décisions prises
- Architecture DDD + Vertical Slice retenue pour tous les services
- PostgreSQL comme base de données de référence (chaque service a la sienne)
- Soft delete avec `deletedAt` pour les entités supprimables
- Préfixe `Port` pour les interfaces de repositories (hexagonal)
- Tous les commentaires de code en anglais
- Documentation du vault en français
